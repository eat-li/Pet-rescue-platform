## P5 技术难点与解决方案（45秒）

### 难点一：购物车批量预约功能

**问题背景**：
- 原系统仅支持单条服务预约
- 用户需要多次填写预约信息，体验差

**解决方案**：
1. **后端设计**：新增 `BatchCreateBookingService` 接口
2. **事务处理**：保证批量创建的一致性
3. **自动清理**：预约成功后自动删除购物车对应项

**核心代码**：
```javascript
// 事务处理保证数据一致性
const bookings = await Promise.all(
  services.map(service => ServiceBooking.create({...}))
)
await CartItem.destroy({
  where: { userId, serviceId: { [Op.in]: serviceIds } }
})
```

---

### 难点二：Sequelize关联查询字段遗漏

**问题现象**：
- 购物车服务状态判断失效，所有服务显示"已下架"
- 鼠标悬停禁用，无法操作

**排查过程**：
```javascript
// 问题代码：attributes 缺少 status 字段
attributes: ['id', 'name', 'type', 'price', 'image', 'weight']
// 修复后：
attributes: ['id', 'name', 'type', 'price', 'image', 'weight', 'status', 'content']
```

**经验总结**：Sequelize关联查询需显式声明所有需要的字段

---

### 难点三：Vue3响应式对象更新

**问题现象**：
- 购物车勾选状态更新后，总价未重新计算

**解决方案**：
- 使用 `computed` 属性实现响应式计算
- 状态变更触发自动重新计算

## 附录：可能的答辩问题

### Q1：为什么选择Vue3而不是Vue2？
**A**：Vue3的组合式API让逻辑复用更方便，性能更好（Tree-shaking、Proxy响应式），且Pinia状态管理比Vuex更简洁。

### Q2：批量预约的事务处理如何保证一致性？
**A**：使用Promise.all并行创建预约记录，全部成功后再删除购物车项。如果中间失败，已创建的预约记录需要回滚（可优化为数据库事务）。

### Q3：前后端如何联调？遇到过什么问题？
**A**：使用axios封装请求，配置baseURL和拦截器。主要问题是跨域和字段类型不匹配（如MySQL的BOOLEAN返回整数）。

### Q4：项目的创新点在哪里？
**A**：
1. 购物车批量预约功能（原系统不支持）
2. 一站式服务整合（领养+服务预约）
3. 完整的状态流转设计（待确认→已确认→已完成）

### Q5：如果用户量增大，系统如何优化？
**A**：
1. 数据库：添加索引、读写分离
2. 后端：Redis缓存、接口限流
3. 前端：懒加载、虚拟滚动
4. 部署：Docker容器化、Nginx负载均衡
