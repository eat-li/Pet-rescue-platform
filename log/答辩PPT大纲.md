### 技术难点与解决方案

**难点**：购物车如何支持多服务批量预约？

**方案对比**：

| 方案 | 描述 | 优劣 |
|------|------|------|
| A ✅ | 后端新增批量预约接口，一次提交多个服务 | 效率高、事务一致 |
| B ❌ | 前端循环调用单个预约接口 | 请求多、无法保证原子性 |

**选型理由**：方案A一次请求完成批量操作，后端可保证**事务一致性**

### 核心代码 — 后端批量预约接口

```javascript
// ServiceBookingService.js — 批量预约核心逻辑
exports.BatchBookingService = async (req, res) => {
  const { items } = req.body  // [{ serviceId, date, timeSlot }]
  const transaction = await sequelize.transaction()
  try {
    const bookings = await Promise.all(
      items.map(item => ServiceBooking.create({
        ...item, userId: req.userId, status: 'pending'
      }, { transaction }))
    )
    await transaction.commit()  // 全部成功才提交
    return res.status(201).json({ code: 201, data: bookings })
  } catch (err) {
    await transaction.rollback()  // 任一失败则回滚
    return res.status(500).json({ code: 500, message: '批量预约失败' })
  }
}
```

> **亮点**：使用 Sequelize 事务（transaction），确保批量操作**原子性**——要么全部成功，要么全部回滚

## 第6页：核心技术难点（60秒）⭐⭐ 重点

### 难点1：图片存储迁移至阿里云OSS

**问题**：本地存储图片导致服务器磁盘压力大、无CDN加速

**解决方案**：迁移至阿里云OSS

```
【修改前】文件存本地，返回相对路径
  前端 ← /avatar/xxx.jpg ← 后端 ← 本地static目录

【修改后】文件存OSS，返回完整URL
  前端 ← https://oss.xxx/avatar/xxx.jpg ← 后端 ← 阿里云OSS
```

**关键改动**：
- Multer 从 `diskStorage` 改为 `memoryStorage`（文件暂存内存）
- 新增 `ossUpload.js` 工具，封装 OSS 上传/批量上传/删除
- 数据库字段从 `VARCHAR(100)` 扩展到 `VARCHAR(500)` 支持 OSS 完整URL

```javascript
// ossUpload.js — OSS上传核心逻辑
async function uploadToOSS(fileBuffer, originalname, folder) {
  const ext = path.extname(originalname) || '.jpg'
  const fileName = `${folder}/${uuid()}${ext}`
  const result = await ossClient.put(fileName, fileBuffer)
  return result.url  // 返回完整OSS URL
}
```

> **影响范围**：7个Service + 7个路由文件，前端**零修改**

### 难点2：401自动登出机制

**问题**：Token过期后，前端仅报错但不自动登出

**解决方案**：Axios响应拦截器统一处理401

```javascript
// http.js — 401统一处理
if (error.response?.status === 401) {
  if (!isHandling401) {
    isHandling401 = true  // 防抖标记
    userStore.clearAuth()  // 清除认证状态
    alert('登录已过期，请重新登录')
    window.location.href = '/login'  // 跳转登录页
  }
}
```

> **双端同步实现**：用户端和管理端均添加，确保任一端Token过期都自动登出

### 难点3：首页API并行化优化

**问题**：首页6个API串行请求，加载耗时过长

**解决方案**：`Promise.all` 并行请求 + `provide/inject` 数据共享

```javascript
// 修改前：6次串行请求，总耗时 ≈ 6 × 单次耗时
const r1 = await getNoticeListAPI()
const r2 = await getArticleListAPI()
// ... 依次等待

// 修改后：并行请求，总耗时 ≈ 单次最长耗时
const [r1, r2, r3, r4, r5, r6] = await Promise.all([
  getNoticeListAPI(), getArticleListAPI(), getAdoptionListAPI(),
  getServiceListAPI(), getPetCountAPI(), getUserCountAPI()
])
provide('homeData', { notices: r1, articles: r2, ... })
```

> **性能提升**：首屏API请求时间减少约 **60-70%**

## 附录：高频答辩问题准备

### Q1：为什么选择Vue而不是React？
> Vue3的组合式API更贴合本项目需求，学习曲线平缓，配合DaisyUI和Element-Plus能快速构建双端差异化UI。同时Vue的响应式系统天然适合数据密集型应用。

### Q2：JWT的Token安全性如何保障？
> 三重保障：(1) Token设置过期时间（24小时），过期自动登出；(2) 使用bcryptjs对密码加密存储，数据库不存明文；(3) 后端中间件统一鉴权，每次请求验证Token合法性。

### Q3：批量预约如何保证数据一致性？
> 使用Sequelize的事务机制（transaction），所有预约操作在同一事务内执行。任一服务预约失败，整个事务回滚，确保不会出现部分预约成功的不一致状态。

### Q4：为什么图片存储从本地迁移到OSS？
> 三个原因：(1) 服务器磁盘空间有限，图片会持续增长；(2) OSS自带CDN加速，图片加载更快；(3) 前后端分离部署时，本地存储的图片路径会失效，OSS用完整URL无此问题。

### Q5：数据库设计中有哪些关联关系？
> 主要关联：User(1)→Pet(N)、User(1)→Article(N)、User(1)→Adoption(N)、Pet(1)→Adoption(1)、Article(1)→Comment(N)、PetServiceOrder(1)→ServiceBooking(N)、User(1)→CartItem(N)。使用Sequelize的belongsTo/hasMany定义关联。

### Q6：购物车的批量预约是如何实现的？
> 后端提供专门的批量预约接口，接收服务ID数组，在Sequelize事务中批量创建预约记录。前端购物车组件支持多选，一键提交，避免多次网络请求。

### Q7：前后端如何保证状态同步（如评论状态）？
> 采用"写入即更新"策略：用户点赞/收藏后，后端同时更新计数字段，前端立即更新本地状态（乐观更新）。管理端修改评论状态后，用户端下次请求即获取最新状态。

### Q8：项目部署方案是什么？
> Nginx反向代理：托管前端静态文件，API请求代理到Node.js后端。后端使用PM2进程管理，保证服务持续运行。数据库使用MySQL，图片使用阿里云OSS。
